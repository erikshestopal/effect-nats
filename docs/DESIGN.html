effect-nats: an Effect-native client for NATS

Design document · draft for review · 2026-07-07 · rev 2 (full surface: core + JetStream + KV + ObjectStore + Services)

Sources of truth: repos/nats.js (client v3 monorepo) and repos/effect-v4 (Effect v4 pre-release).

1. The problem: two resource-management worlds

NATS ships an excellent TypeScript client. It manages a socket, speaks the wire protocol, reconnects, and hands you messages. It expresses all of this in the standard vocabulary of JavaScript: promises for one-shot operations, async iterables for message flows, thrown Error subclasses for failure, and manual close()/drain() calls for cleanup.

An Effect program has a different vocabulary for exactly the same concerns. Failures are values in a typed error channel, not thrown exceptions. Anything that must be cleaned up is tied to a Scope, so cleanup happens even when a fiber is interrupted. Long-lived flows of values are Streams, which compose with retry policies, batching, and concurrency operators. Dependencies arrive through the context as services provided by Layers, so a test can swap the real connection for a fake one without touching call sites.

When you use the raw NATS client inside an Effect program, you sit at the seam between these two worlds and hand-translate at every call site: Effect.tryPromise here, an acquireRelease there, an untyped catch block guessing which of eleven error classes you got. Every service that touches NATS re-invents this translation, slightly differently, with slightly different bugs. The translation is mechanical, which is precisely why it should be written once, in one library, and never again.

That is the entire ambition of this design: a thin, total translation layer over the whole client surface — the core connection, JetStream (persistence), KV, ObjectStore, and the Services framework. We do not reimplement the protocol parser, the reconnect loop, or the consumer bookkeeping — the vendored @nats-io/* packages already do those things well and are maintained by the NATS team. We wrap their public surfaces so that, from the Effect side, NATS looks like it was born there. The precedent inside Effect itself is @effect/ai-anthropic, which wraps the Anthropic HTTP API behind a Context.Service without re-deriving HTTP, and effect/unstable/socket, which defines one socket interface and lets each runtime provide it as a layer.

The cost of wrapping instead of rewriting: we inherit the SDK's operational behavior wholesale — its buffering decisions, its reconnect semantics, its module-global transport registry (§4 prices that one honestly). Where the SDK's behavior is un-Effect-like, we can only paper over it, not fix it. We accept that trade because the alternative — maintaining a protocol implementation — is a full-time job the NATS team is already doing.

2. NATS in one section

NATS is a message router. A client opens one TCP or WebSocket connection to a server and sends messages labeled with a subject — a dot-separated string like orders.eu.created. Other clients declare interest in subjects, with wildcards (orders.*.created, orders.>). The server copies each incoming message to every matching interest. That declared interest is a subscription.

Calibrate against Kafka, since that is the mental model most engineers carry: NATS core is Kafka minus the log, minus partitions, minus consumer groups, minus retention. Messages are not stored; if nobody is subscribed at the instant a message passes through the server, it is gone. What you get in exchange is sub-millisecond fan-out, no broker state to administer per topic (subjects are just strings, never "created"), and millions of subjects for free. The missing log is bolted back on by a layer called JetStream — in scope for this design, derived properly in §11.

Two conventions built from these primitives matter for our API surface. First, if several subscribers join the same subject under a shared queue group name, the server delivers each message to exactly one of them — load balancing without a coordinator. Second, a publisher can attach a reply subject to a message; a responder publishes its answer to that subject, and the requesting client subscribes to it beforehand. That publish-plus-ephemeral-subscription pattern is request/reply, and the SDK wraps it in a single request() call. The server even short-circuits the failure case: if a request goes to a subject with no subscribers, the server immediately sends back a 503-status message, which the SDK surfaces as NoRespondersError instead of letting you wait out a timeout.

  publisher                     nats-server                    subscribers
  ---------                     -----------                    -----------
  PUB orders.eu.created  ───▶   match "orders.*.created" ───▶  svc-a (queue "billing")
  (payload: 143 bytes)          match "orders.>"        ───▶   svc-b
                                                        ───▶   svc-c (queue "billing")
                                one of {svc-a, svc-c} gets it (queue group);
                                svc-b always gets it (plain subscription);
                                nothing is stored anywhere

The consequence for our design: the core client object is small. One connection, four verbs — publish, subscribe, request, drain — plus connection-lifecycle observation. Everything else in this document (JetStream, KV, ObjectStore, Services) is built by the upstream SDK on top of these verbs, over the same single connection — which is why our wrapper composes the same way, layer by layer.

4. The connection as a scoped service

4.1 Why a service, and which idiom

A NATS connection is expensive (TCP handshake, INFO/CONNECT exchange, optional TLS upgrade), long-lived, and shared by everything in the process that talks to NATS. That is the textbook profile of an Effect service: constructed once, placed in the context, torn down when its scope closes. Effect v4's idiom for this is the class-style Context.Service — the class itself is the tag, is yieldable, and carries .of() — with construction exposed as make (an effect) and provision as layer / layerConfig. This is letter-for-letter the shape of AnthropicClient in repos/effect-v4/packages/ai/anthropic:

export class NatsClient extends Context.Service<NatsClient, Service>()(
  "effect-nats/NatsClient"
) {}

export const make: (options?: Options) =>
  Effect.Effect<Service, ConnectionError | TimeoutError, NatsConnector | Scope.Scope>

export const layer: (options?: Options) =>
  Layer.Layer<NatsClient, ConnectionError | TimeoutError, NatsConnector>

Acquisition wraps the SDK's connect() promise in Effect.acquireRelease. Release runs drain() — the SDK's ordered shutdown, which finishes in-flight messages before closing — falling back to close() if draining fails or exceeds a bounded timeout. Interruption of any fiber holding the scope therefore cannot leak a socket, which is the property the raw SDK cannot give you.

4.2 Transports: derive the design from the SDK's one global

Here the SDK has a sharp edge we must design around rather than inherit. nats.js selects its transport through a module-global factory: connect() from @nats-io/transport-node calls setTransportFactory(tcp) and then dials; wsconnect() from core calls setTransportFactory(websocket) and then dials (core/src/ws_transport.ts:354, transport-node/src/connect.ts). Both funnel into the same NatsConnectionImpl.connect. So "which transport" is not a per-connection option — it is ambient mutable state, decided by which entry function you happened to import.

We make that decision explicit and typed: a second, tiny service whose only job is to know how to dial.

export class NatsConnector extends Context.Service<NatsConnector, Connector>()(
  "effect-nats/NatsConnector"
) {}

export interface Connector {
  readonly connect: (options: ConnectionOptions) =>
    Effect.Effect<NatsConnection, ConnectionError | TimeoutError>
}

// in NatsConnector.ts — zero extra dependencies, wsconnect ships in nats-core.
// Uses globalThis.WebSocket (Node ≥ 22 stable, Bun, browsers); pass wsFactory
// via transformOptions for runtimes without it or for custom headers.
export const layerWebSocket: Layer.Layer<NatsConnector>

// in NodeConnector.ts — raw TCP via @nats-io/transport-node. Node and Bun.
// Lives in its own module file so WebSocket-only bundles never load node:net.
export const layer: Layer.Layer<NatsConnector>

This mirrors effect/unstable/socket exactly: one abstract interface in the shared module, one layer per runtime (NodeSocket.layerNet, BunSocket, Socket.layerWebSocket). An application composes:

const NatsLive = NatsClient.layer({ servers: "nats://10.0.0.7:4222" }).pipe(
  Layer.provide(NodeConnector.layer)        // or NatsConnector.layerWebSocket
)

        ┌──────────────────────────────┐
        │        application           │
        │   yield* NatsClient          │
        └──────────────┬───────────────┘
                       │ requires
        ┌──────────────▼───────────────┐
        │  NatsClient.layer(options)   │   scoped: acquire = connect()
        │                              │           release = drain() → close()
        └──────────────┬───────────────┘
                       │ requires
        ┌──────────────▼───────────────┐
        │        NatsConnector         │   one of:
        ├──────────────────────────────┤
        │ NodeConnector.layer          │──▶ @nats-io/transport-node  (TCP, node/bun)
        │ NatsConnector.layerWebSocket │──▶ wsconnect / nats-core    (WS, node≥22/bun/browser)
        └──────────────────────────────┘

The honestly-priced failure mode: the global factory is still global underneath us. Two fibers concurrently building one TCP-backed client and one WebSocket-backed client in the same process can interleave setTransportFactory calls and dial the wrong transport. The wrapper serializes connection establishment behind a module-level Semaphore (Effect has one; effect/Semaphore) so the set-then-dial pair is atomic within our library. Code that bypasses us and calls the SDK directly can still race us. We document this rather than pretend to fix it; fixing it for real is an upstream change.

4.3 The escape hatch

The service exposes the raw connection as readonly connection: NatsConnection. Every wrapper that hides its SDK eventually strands a user one missing feature away from forking it; the Anthropic wrapper exposes its generated client for the same reason. Using the escape hatch means leaving typed-error land — that is the visible, priced cost, and it is the right default for the 1% of calls we don't cover (publishMessage, reconnect(), setServers, stats plumbing).

5. Configuration

The SDK's ConnectionOptions is a flat bag of ~25 fields with three problems from an Effect standpoint: secrets are plain strings (pass, token), durations are bare millisecond numbers (timeout, pingInterval, reconnectTimeWait), and the four mutually-exclusive authentication mechanisms (user/pass, token, creds file, nkey/JWT) are expressed as optional fields whose invalid combinations fail at runtime.

Our Options type fixes each with the module Effect already ships for it, and nothing more:

export type Options = {
  readonly servers?: string | ReadonlyArray<string>
  readonly name?: string
  readonly auth?: Auth                       // tagged union, below
  readonly timeout?: Duration.Input          // @default "20 seconds"
  readonly pingInterval?: Duration.Input     // @default "2 minutes"
  readonly maxPingOut?: number
  readonly reconnect?: boolean | ReconnectOptions
  readonly noEcho?: boolean
  readonly inboxPrefix?: string
  readonly tls?: TlsOptions
  readonly ignoreClusterUpdates?: boolean
  // escape hatch, mirroring AnthropicClient's transformClient:
  readonly transformOptions?: ((options: ConnectionOptions) => ConnectionOptions)
}

export type ReconnectOptions = {
  readonly maxAttempts?: number              // -1 = forever
  readonly wait?: Duration.Input             // @default "2 seconds"
  readonly jitter?: Duration.Input
  readonly waitOnFirstConnect?: boolean
}

export type Auth =
  | { readonly _tag: "UserPass"; readonly user: string; readonly pass: Redacted.Redacted<string> }
  | { readonly _tag: "Token";    readonly token: Redacted.Redacted<string> }
  | { readonly _tag: "Creds";    readonly creds: Redacted.Redacted<string> }  // .creds file contents
  | { readonly _tag: "NKey";     readonly seed: Redacted.Redacted<string> }

The tagged union is translated internally to the SDK's credsAuthenticator / nkeyAuthenticator / tokenAuthenticator / usernamePasswordAuthenticator (all exported from nats-core), with Redacted.value unwrapped at the last possible moment. A constructor per variant (NatsClient.authToken({ token }), …) keeps call sites free of _tag literals.

A note on optionality, since it is a project conviction: Option is for model positions — data flowing through the program whose absence a consumer branches on (NatsMessage.replyTo, NatsKv.get). Options bags are input positions where absence means "use the default". This repository enables exactOptionalPropertyTypes and forbids nullish unions via the no-nullish-types ast-grep rule, so the normative spelling — used throughout this document and in the implementation — is plain readonly field?: T, and it means strictly "present or absent": explicitly passing undefined is a type error. (Upstream Effect v4 instead spells these ?: T | undefined to permit explicit undefined; we deliberately take the stricter line.) Conditional option assembly therefore uses conditional spreads (...(Predicate.isNotUndefined(x) ? { timeout: x } : {})), and undefined never appears in the public surface — nor do null or nullish unions, anywhere.

Alongside layer(options) for explicit values, layerConfig takes Config values and defers to the same make — the two-tier convention every Effect provider package follows:

const NatsLive = NatsClient.layerConfig({
  servers: Config.String("NATS_SERVERS").pipe(Config.map((s) => s.split(","))),
  auth: Config.Redacted("NATS_TOKEN").pipe(Config.map((token) => NatsClient.authToken({ token })))
}).pipe(Layer.provide(NodeConnector.layer))

Trade-off: our Options deliberately omits low-level fields (pedantic, verbose, noAsyncTraces, reconnectDelayHandler, …). Anything omitted remains reachable through transformOptions. The price is that the escape hatch takes raw milliseconds and raw strings — SDK types, undecorated. That is intentional: the hatch should feel like a hatch.

6. Errors as values

The SDK defines its core error classes in core/src/errors.ts and signals failure by throwing or rejecting. Callers discriminate with instanceof — invisible to the type system, so nothing tells you that request() can fail with NoRespondersError until production does. Effect's error channel exists precisely to make that set visible and exhaustive.

Effect v4's idiom for framework-grade errors is Schema.TaggedErrorClass (see SocketError and friends in unstable/socket/Socket.ts): tagged for catchTag, schema-backed so errors serialize across process boundaries, with opaque causes carried as Schema.Defect:

export class NoRespondersError extends Schema.TaggedErrorClass<NoRespondersError>(
  "effect-nats/NatsError/NoRespondersError"
)("NoRespondersError", {
  subject: Schema.String
}) {}

We mirror the SDK taxonomy one-to-one — a mapping function at the boundary turns each instanceof into the corresponding tagged class, and anything unrecognized becomes a defect (a bug in our mapping, not a domain failure the caller should handle):

Tagged errorCarriesRaised by

TimeoutError—connect, request, flush, rtt

DrainingConnectionError—any op while draining
NoRespondersErrorsubjectrequest / requestMany

AuthorizationError—connect, server auth revocation
PermissionViolationErroroperation, subject, queue: Optionpub/sub against server ACLs

Two SDK classes are deliberately not mirrored: InvalidOperationError (e.g. iterating a callback-mode subscription) and InvalidArgumentError (malformed options). Both indicate a bug in the calling code or in this wrapper, not a condition a caller should branch on, so they surface as defects. Our typed Options makes most InvalidArgumentError cases unrepresentable in the first place; the residual path is transformOptions, which is a hatch and priced as one.

One implementation note that must survive into code review: the boundary mapper matches on instanceof, never on .name — the SDK's NoRespondersError sets this.name = "NoResponders", not "NoRespondersError". JetStream contributes its own error hierarchy, mapped the same way in its own module (§11.5).

export type NatsError =
  | ConnectionError | TimeoutError | ClosedConnectionError | DrainingConnectionError
  | NoRespondersError | RequestError | AuthorizationError | PermissionViolationError
  | ProtocolError | InvalidSubjectError | UserAuthenticationExpiredError | NoReplySubjectError

export const isNatsError: (u: unknown) => u is NatsError

Design choice, priced: the socket module takes the other road — one root SocketError wrapping a reason union. That keeps method signatures short at the cost of two-level pattern matching (catchTag("SocketError"), then match on reason). We choose the flat union and give each operation its narrowest error type (request fails with TimeoutError | NoRespondersError | RequestError | ClosedConnectionError | DrainingConnectionError | InvalidSubjectError; publish can never fail with NoRespondersError). Flat unions make Effect.catchTag("NoRespondersError", …) a one-liner, which is the single most common thing users do with NATS errors. The price is longer signatures and the risk that adding a new failure mode to an operation is a breaking change to its type — a price we accept because that break is exactly the information the caller needs.

7. Messages and headers

The SDK's Msg has three habits that don't survive contact with Effect code: reply is string | undefined (every consumer writes the same if), json() throws on malformed payloads, and headers is an optional, mutable, throwing-getter MsgHdrs. None of these needs new machinery — Effect has Option, Schema, and immutable-view patterns for exactly these shapes.

export interface NatsMessage {
  readonly [TypeId]: typeof TypeId
  readonly subject: string
  readonly payload: Uint8Array
  readonly replyTo: Option.Option<string>
  readonly headers: NatsHeaders.NatsHeaders          // empty view when absent
}

// module functions (dual where a data-last pipe form is useful):
export const text: (self: NatsMessage) => string
export const schemaJson: <S extends Schema.Top>(schema: S) =>
  (self: NatsMessage) => Effect.Effect<S["Type"], Schema.SchemaError>
export const respond: {
  (options?: RespondOptions): (self: NatsMessage) => Effect.Effect<void, NoReplySubjectError | ClosedConnectionError>
  (self: NatsMessage, options?: RespondOptions): Effect.Effect<void, NoReplySubjectError | ClosedConnectionError>
}

schemaJson is composition, not code: Schema.fromJsonString(schema) already exists in v4, so decoding a JSON payload is "decode bytes to string, then run the transformed schema" — and the error is a real SchemaError carrying the path to the offending field, not a bare SyntaxError. The wrapper object is a lazy view over the SDK Msg (which it retains privately so respond can delegate); we allocate one small object per message and defer header/reply materialization until asked. That per-message allocation is the cost of immutability here, and at NATS message rates it is measurable — if profiling ever shows it, the escape hatch (subscribeRaw, §8) exists. The same wrapper strategy — and the same schemaJson/text helpers, shared internally — reappears for JetStream's JsMsg (§11.4) and KV's KvEntry (§12), so payload decoding is one idea across the whole library.

NatsHeaders is the same story in miniature: an immutable view over MsgHdrs with get returning Option<string>, getAll returning ReadonlyArray<string>, and a fromInput constructor accepting a plain record for the publish path. We do not adopt effect/unstable/http's Headers module: NATS headers are case-sensitive by default with canonical-MIME matching as an opt-in (Match in the SDK), and HTTP semantics would quietly corrupt that.

8. Subscriptions are streams

A subscription is a flow of messages that begins when interest is declared and must end with an explicit unsubscribe() or drain() — cleanup attached to a flow of values. In Effect that is the definition of a scoped Stream, so the translation is almost embarrassingly direct: the SDK's Subscription already implements AsyncIterable<Msg>, and v4 ships Stream.fromAsyncIterable(iterable, onError).

readonly subscribe: (subject: string, options?: SubscribeOptions) =>
  Stream.Stream<NatsMessage.NatsMessage, NatsError>

export type SubscribeOptions = {
  readonly queue?: string      // queue-group load balancing (§2)
  readonly max?: number        // auto-unsubscribe after n messages
}

Construction: Effect.acquireRelease creates the SDK subscription and registers drain() (bounded, falling back to unsubscribe()) as the finalizer; the acquired subscription is unwrapped into the stream via Stream.fromAsyncIterable, mapping each Msg to a NatsMessage and each iterator error through the §6 boundary mapper. When a consumer interrupts the stream — a Stream.take(5), a failed downstream stage, fiber interruption — the scope closes and the server-side interest is removed. Forgetting to unsubscribe stops being a possible bug.

  Effect scope                         SDK                         server
  ────────────                         ───                         ──────
  stream started ───────────────────▶  nc.subscribe("orders.>")──▶ +interest
       │                                    │
       │  pull ◀── for-await ◀── QueuedIterator ◀── socket bytes
       │  pull ◀── ...                      │
       ▼                                    ▼
  scope closes ────────────────────▶  sub.drain() ───────────────▶ -interest
  (take(n) done / error / interrupt)  (in-flight msgs processed first)

Backpressure, honestly: for core subscriptions there is none end-to-end, and we must not imply otherwise. The NATS server pushes; the SDK's QueuedIterator buffers unboundedly in process memory when your consumer is slower than the publisher. Our stream pulls from that buffer, so Effect-side operators (Stream.buffer, throttle, mapEffect with bounded concurrency) control everything downstream of the buffer but nothing upstream of it. The SDK's own pressure valve is detection, not prevention: pass the experimental slow threshold and watch for slowConsumer status events (§10). The real answers are queue groups for load distribution and — now in scope — JetStream pull consumers (§11.3), whose batch-request protocol is genuine, server-honored flow control.

Two deliberate omissions. The SDK's callback-mode subscription (SubscriptionOptions.callback) exists to skip the iterator's microtask overhead; we do not expose it in v1 because Stream.callback + Queue.offerUnsafe gives us the same fast path if benchmarks demand it later without changing the public type — the constructor swap is invisible behind Stream. And subscribeRaw (same stream, but of raw SDK Msg) is reserved as the documented performance hatch, consistent with §4.3.

9. Request/reply

A request is a one-shot effect with a deadline and a known failure vocabulary — the shape Effect models natively:

readonly request: (subject: string, options?: RequestOptions) =>
  Effect.Effect<
    NatsMessage.NatsMessage,
    TimeoutError | NoRespondersError | RequestError | InvalidSubjectError
      | ClosedConnectionError | DrainingConnectionError
  >

export type RequestOptions = {
  readonly payload?: Payload                 // Uint8Array | string
  readonly headers?: NatsHeaders.Input
  readonly timeout?: Duration.Input          // @default "1 second" (SDK default)
}

NoRespondersError deserves its place in the signature: it is the difference between "the service is down" (immediate, actionable, retry elsewhere) and "the service is slow" (TimeoutError), and the server tells us which within one round-trip. Typed errors make the distinction impossible to ignore; with the raw SDK both arrive as rejected promises and are routinely conflated.

One SDK limitation passes through: nats.js's request() takes no abort signal, so when an Effect fiber is interrupted mid-request we abandon the promise rather than cancel the server-side interest. The muxed inbox subscription makes this cheap (one pending entry times out internally), but "interrupt = abandon, not cancel" is the semantic and the JSDoc will say so.

requestMany — scatter/gather over a subject with multiple responders — is a bounded flow, i.e. a Stream. The SDK's termination strategies map onto one options object; maxWait is the only required bound:

readonly requestMany: (subject: string, options: RequestManyOptions) =>
  Stream.Stream<NatsMessage.NatsMessage, NatsError>

export type RequestManyOptions = {
  readonly payload?: Payload
  readonly headers?: NatsHeaders.Input
  readonly maxWait: Duration.Input                       // hard deadline for the whole gather
  readonly maxMessages?: number              // stop after n replies ("count")
  readonly stall?: Duration.Input            // stop after a quiet gap ("stall")
  readonly sentinel?: boolean                // stop on an empty-payload end marker ("sentinel")
}

The SDK selects among four named strategies ("timer" | "count" | "stall" | "sentinel", core/src/request.ts); we derive the strategy from which bound the caller supplies instead of asking them to name it, with maxWait always the hard ceiling.

10. Connection lifecycle as observable data

Reconnection is the SDK's crown jewel and we do not touch it: server-pool gossip, exponential-ish delays with jitter, auth-failure abort — all inherited as-is, tuned through §5's reconnect options. What the SDK reports about that machinery is an AsyncIterable<Status> — an eleven-variant discriminated union (disconnect, reconnecting, reconnect, update, ldm, ping, staleConnection, slowConsumer, forceReconnect, error, close). We expose it as an infinite, never-failing stream and re-export the union as ConnectionStatus:

readonly status: Stream.Stream<ConnectionStatus>     // multiple consumers allowed;
                                                     // each subscribe() on the SDK side
readonly closed: Effect.Effect<Option.Option<NatsError>>  // resolves once, at close;
                                                            // Some(e) when close was caused by e

// typical use: structured logging of connection health
yield* nats.status.pipe(
  Stream.runForEach((s) => Effect.logInfo("nats", s)),
  Effect.forkScoped
)

The SDK's closed() promise "resolves to an error instead of rejecting" — a promise-land contortion for "tell me how it ended without forcing a try/catch". Effect.Effect<Option.Option<NatsError>> says the same thing in the type: it always succeeds, and the payload tells you whether the ending was clean. One leak we accept for v1: the slowConsumer variant carries a raw SDK Subscription object, because remapping it to our stream handles would require bookkeeping disproportionate to its diagnostic value.

11. JetStream: the log, bolted back on

11.1 Why it exists, and what the words mean

Core NATS forgets. A message that finds no subscriber at the instant it transits the server is gone, and a subscriber that crashes loses whatever was in flight. To promise "this order event will be processed at least once" you need two things the core protocol lacks: the server must store messages, and it must track which ones each reader has confirmed. JetStream is that layer, built by the NATS server on top of the same wire protocol. A stream is a named, stored, subject-filtered log — here the Kafka comparison finally inverts: this is the log, with retention limits, replicas, and sequence numbers. A consumer is a server-side cursor over a stream: it remembers a position, redelivers messages that were not acknowledged within ack_wait, and counts delivery attempts.

The delivery mechanism worth understanding, because our whole streaming story hangs on it, is the pull protocol. The client sends the server a bounded request — "give me up to 100 messages or 1 MB, expiring in 30 seconds" — and the server sends at most that much. The client acknowledges each message individually and asks again when it wants more. Contrast §8: core subscriptions are a firehose the client can only buffer; pull consumers are a faucet the client opens by measured amounts. This is genuine, server-honored backpressure, and it is the reason JetStream belongs in v1 rather than a roadmap.

11.2 The services

The SDK splits data plane from admin plane — jetstream(nc) returns the publishing/consuming client, jetstreamManager(nc) the stream/consumer CRUD API — and we keep that split as two services in two modules, both layered on NatsClient:

export class JetStream extends Context.Service<JetStream, Service>()(
  "effect-nats/JetStream"
) {}
export const layer: (options?: JetStreamOptions) =>
  Layer.Layer<JetStream, never, NatsClient>

export class JetStreamManager extends Context.Service<JetStreamManager, Service>()(
  "effect-nats/JetStreamManager"
) {}
export const layer: (options?: JetStreamOptions) =>
  Layer.Layer<JetStreamManager, never, NatsClient>

export type JetStreamOptions = {
  readonly apiPrefix?: string
  readonly domain?: string
  readonly timeout?: Duration.Input
}

Publishing to a stream differs from core publish in one fundamental way: the server answers. The reply — which stream stored it, at what sequence, whether it was a duplicate — is a PubAck, and the options carry the deduplication and optimistic-concurrency machinery (msgID, expect.lastSubjectSequence, …) that KV is later built from:

readonly publish: (subject: string, options?: PublishOptions) =>
  Effect.Effect<PubAck, JetStreamError | WrongLastSequenceError | TimeoutError | JetStreamNotEnabledError>

export type PublishOptions = {
  readonly payload?: Payload
  readonly headers?: NatsHeaders.Input
  readonly msgID?: string                // server-side dedup within duplicate_window
  readonly timeout?: Duration.Input
  readonly retries?: number              // on no-responders/timeout; SDK default 1
  readonly expect?: {                                // optimistic concurrency, all optional:
    readonly streamName?: string
    readonly lastMsgID?: string
    readonly lastSequence?: number
    readonly lastSubjectSequence?: number
  }
}

export type PubAck = {
  readonly stream: string
  readonly seq: number
  readonly duplicate: boolean
}

11.3 Consuming: three verbs, three shapes

The SDK gives a consumer three ways to receive, and each maps onto a different Effect shape because each has different termination semantics (verified in jetstream/src/consumer.ts):

SDK verbSemanticsEffect shape
next()one message or null after expiresEffect<Option<JsMessage>, …>
fetch(opts)one bounded pull; iterator self-terminates when the batch/byte/time budget is spent; missed heartbeats end it with an errorbounded Stream<JsMessage, JetStreamError>
consume(opts)endless: SDK re-issues pull requests whenever the buffered count falls to 75% of max_messages; never self-terminates; self-heals through heartbeat lossinfinite Stream<JsMessage, JetStreamError>, ended only by scope close or fatal protocol error
readonly consumer: (stream: string, name?: string | OrderedConsumerOptions) =>
  Effect.Effect<JsConsumer, StreamNotFoundError | ConsumerNotFoundError | JetStreamApiError>

export interface JsConsumer {
  readonly next: (options?: NextOptions) =>
    Effect.Effect<Option.Option<JsMessage.JsMessage>, JetStreamError>
  readonly fetch: (options?: FetchOptions) =>
    Stream.Stream<JsMessage.JsMessage, JetStreamError>
  readonly consume: (options?: ConsumeOptions) =>
    Stream.Stream<JsMessage.JsMessage, JetStreamError>
  readonly info: (cached?: boolean) => Effect.Effect<ConsumerInfo, JetStreamApiError>
}

export type ConsumeOptions = {
  readonly maxMessages?: number            // buffered pull size, SDK default 100
  readonly maxBytes?: number
  readonly thresholdMessages?: number      // re-pull trigger, default 75%
  readonly abortOnMissingResource?: boolean
  readonly bind?: boolean
  // side-channel for the SDK's 15-variant ConsumerNotification union —
  // heartbeats_missed, ordered_consumer_recreated, consumer_deleted, …
  readonly onNotification?: ((n: ConsumerNotification) => Effect.Effect<unknown>)
}

Omitting the consumer name requests an ordered consumer — an ephemeral, auto-recreated cursor that guarantees in-order delivery by tearing itself down and resuming from the last known sequence whenever it detects a gap. The SDK handles the recreation invisibly (emitting ordered_consumer_recreated notifications); our stream simply keeps flowing, which is exactly the semantics a stream consumer wants.

Notifications deserve their design note. A consume() session carries a second async iterable, status(), with fifteen informational variants. They are diagnostics, not data — mixing them into the message stream would force every consumer to filter them back out, and giving them a separate first-class stream would double the surface for something most callers ignore. The onNotification handler — run in a forked, scope-bound fiber — is the proportionate design: invisible when unused, fully typed when needed. The one variant that is not merely informational is priced by the SDK itself: on fetch() (bounded, non-ordered), two missed heartbeats terminate the iterator with an error, and our stream fails with a typed JetStreamError; on consume() the SDK recovers by re-syncing, and we stay silent apart from the notification.

Scope-tied shutdown: the stream's finalizer calls ConsumerMessages.close(), which stops pulling and resolves after in-flight messages are handed over. Combined with ack_wait redelivery, interruption is safe by construction: anything your fiber did not ack before dying comes back to the next consumer instance. That property — interruption-safety for free — is the single biggest win of the whole JetStream wrapper.

11.4 The message and its acknowledgments

A JetStream message is a core message plus a delivery context (stream sequence, delivery count, pending count) and an acknowledgment vocabulary. The wrapper shape follows §7, with acks as effects:

export interface JsMessage extends NatsMessage.NatsMessage {
  readonly stream: string
  readonly consumer: string
  readonly seq: number                        // stream sequence
  readonly deliveryCount: number
  readonly redelivered: boolean               // deliveryCount > 1
  readonly pending: number                    // msgs the server still holds for this consumer
  readonly time: DateTime.Utc                 // effect/DateTime, not a JS Date
}

export const ack:     (self: JsMessage) => Effect.Effect<void>
export const nak:     (self: JsMessage, options?: { readonly delay?: Duration.Input }) => Effect.Effect<void>
export const working: (self: JsMessage) => Effect.Effect<void>      // extend ack_wait
export const term:    (self: JsMessage, options?: { readonly reason?: string }) => Effect.Effect<void>
export const ackAck:  (self: JsMessage) => Effect.Effect<boolean, TimeoutError>  // round-trip confirmed
// text / schemaJson inherited conceptually from NatsMessage (§7)

The four verbs are the consumer's contract with the server: ack ("done, advance the cursor"), nak ("failed, redeliver — optionally after a delay"), working ("still processing, reset my ack_wait clock"), term ("never redeliver this one"). Only ackAck costs a round-trip; the rest are fire-and-forget publishes, which is why they cannot fail with anything but a closed connection (a defect at that point, since the enclosing stream has already failed). A convenience combinator earns its keep here:

// process-and-ack: ack on success, nak on failure, term on defect — the
// 90% loop written once. Handler + one options object, per convention
// (Stream.mapEffect(f, { concurrency }) is the shape, not currying):
export const processWith: <A, E, R>(
  handler: (msg: JsMessage) => Effect.Effect<A, E, R>,
  options?: { readonly nakDelay?: Duration.Input }
) => (msg: JsMessage) => Effect.Effect<void, never, R>

11.5 The manager, listers, and JetStream errors

JetStreamManager wraps the admin plane: streams (add, update, info, delete, purge, getMessage) and consumers (add, update, info, delete, pause, resume). Two design decisions:

Config passthrough. StreamConfig and ConsumerConfig are ~40 snake_case fields each, mirroring server JSON (retention/storage/ack_policy enums, Nanos durations). We pass these through as re-exported SDK types rather than re-modeling them — re-typing 80 fields buys churn, not safety, since the server is the validator. The exception: fields that are durations in milliseconds or nanoseconds get accepted as Duration.Input at our constructors and converted (Duration.toMillis / nanos()), because raw Nanos numbers are the single most error-prone part of the upstream API.

Listers are streams. The SDK's Lister<T> paginates (next(): Promise<T[]> per page, async-iterable per item). Stream.paginate exists for precisely this shape, so streams.list() and consumers.list(stream) return Stream<StreamInfo> / Stream<ConsumerInfo> and pagination disappears from the caller's view.

JetStream errors live in their own module, mapped like §6: JetStreamNotEnabledError, JetStreamError, JetStreamStatusError (code), JetStreamApiError (code = NATS err_code, status = HTTP-ish), StreamNotFoundError (10059), ConsumerNotFoundError (10014), InvalidNameError, and — pulled out by err_code 10071/10164 because callers branch on it — WrongLastSequenceError, the optimistic-concurrency conflict that powers KV's compare-and-set.

Deliberately deferred within JetStream

Push consumers (legacy delivery model; pull is the modern path and the only one new servers optimize), startBatch/FastIngest (atomic batching, server 2.12+/2.14+, API still moving), DirectStreamAPI, scheduled publishes (schedule/cancelSchedule), and advisories(). All remain reachable via the raw-client escape hatch (JetStream.client exposes the SDK JetStreamClient), and none blocks a later additive module.

12. KV: a bucket is a stream wearing a hash-map costume

Once a server can store a log and answer "what is the last message on subject X", a key-value store falls out almost for free. A bucket named orders is a JetStream stream named KV_orders holding one subject per key; put publishes to the key's subject; get reads the subject's latest message; the revision is the stream sequence number; compare-and-set is the expect.lastSubjectSequence header from §11.2; a delete is a tombstone message with operation DEL. History and watch are the stream's own replay and tail. That is the entire mechanism — KV is a projection of JetStream, and the SDK's Kvm class is honest about it (it takes a JetStream client in its constructor). Ours does the same.

// NatsKv.ts — a bucket handle is a resource, not a singleton:
export const open:   (bucket: string) => Effect.Effect<Kv, BucketNotFoundError | JetStreamApiError, JetStream>
export const create: (bucket: string, options?: BucketOptions) => Effect.Effect<Kv, JetStreamApiError, JetStream>
// and for the common one-bucket app, a service + layer over the handle:
export class NatsKv extends Context.Service<NatsKv, Kv>()("effect-nats/NatsKv") {}
export const layer: (bucket: string, options?: BucketOptions) =>
  Layer.Layer<NatsKv, JetStreamApiError, JetStream>

export interface Kv {
  readonly get: (key: string, options?: { readonly revision?: number }) =>
    Effect.Effect<Option.Option<KvEntry>, JetStreamError>
  readonly put: (key: string, value: Payload) =>
    Effect.Effect<number, JetStreamError>                          // new revision
  readonly create: (key: string, value: Payload) =>
    Effect.Effect<number, KeyExistsError | JetStreamError>         // fails if key live
  readonly update: (key: string, value: Payload, options: { readonly revision: number }) =>
    Effect.Effect<number, WrongLastSequenceError | JetStreamError> // CAS
  readonly delete: (key: string) => Effect.Effect<void, JetStreamError>   // tombstone, history kept
  readonly purge:  (key: string) => Effect.Effect<void, JetStreamError>   // history erased
  readonly keys:    (filter?: string) => Stream.Stream<string, JetStreamError>        // bounded
  readonly history: (options?: { readonly key?: string }) =>
    Stream.Stream<KvEntry, JetStreamError>                                            // bounded
  readonly watch:   (options?: WatchOptions) => Stream.Stream<KvEntry, JetStreamError> // infinite
  readonly status:  Effect.Effect<KvStatus, JetStreamError>
}

export interface KvEntry {
  readonly key: string
  readonly value: Uint8Array
  readonly revision: number
  readonly operation: "PUT" | "DEL" | "PURGE"
  readonly created: DateTime.Utc
  readonly isUpdate: boolean     // false during watch catch-up, true once live
}

update's error type is where the derivation pays off: the SDK surfaces a CAS conflict as a JetStream 409 with err_code 10071, which §11.5 already reified as WrongLastSequenceError — so Effect.retry with a fresh get is a typed, five-line optimistic-concurrency loop. create on a live key is the same server rejection; we re-tag it KeyExistsError because "already exists" is what the caller means to catch. Termination semantics follow the SDK exactly: watch() never ends on its own (scope close stops it); keys() and history() end when the catch-up is delivered.

The compatibility dividend: Effect v4 ships a standard store interface, effect/unstable/persistence/KeyValueStore (string/Uint8Array get/set/remove, plus schema stores). A NATS bucket satisfies it trivially, so we also ship:

export const layerKeyValueStore: (bucket: string, options?: BucketOptions) =>
  Layer.Layer<KeyValueStore.KeyValueStore, JetStreamApiError, JetStream>

Any Effect code written against the standard interface — persisted caches, schema stores — runs on NATS KV unchanged. The mapping discards revisions and watch (the standard interface has neither); code that needs those uses Kv directly. Cheap to build, disproportionately useful.

13. ObjectStore: large values, chunked

KV values ride in single messages, and messages have a size ceiling (server default 1 MB). To store a 200 MB artifact you split it into chunks, store each chunk as a message, and keep a manifest — name, size, chunk count, SHA-256 digest — as a separate metadata entry. That is ObjectStore: another JetStream projection, with the manifest called ObjectInfo. The SDK exposes the chunked I/O as web-standard ReadableStream<Uint8Array>, and Effect v4 has the adapters (Stream.fromReadableStream, Stream.toReadableStreamWith), so the wrapper is again composition:

export const open:   (bucket: string) => Effect.Effect<ObjectStore, BucketNotFoundError | JetStreamApiError, JetStream>
export const create: (bucket: string, options?: BucketOptions) => Effect.Effect<ObjectStore, JetStreamApiError, JetStream>
export class NatsObjectStore extends Context.Service<NatsObjectStore, ObjectStore>()(
  "effect-nats/NatsObjectStore"
) {}
export const layer: (bucket: string, options?: BucketOptions) => Layer.Layer<...>

export interface ObjectStore {
  readonly put: (meta: ObjectMeta, data: Stream.Stream<Uint8Array, unknown>) =>
    Effect.Effect<ObjectInfo, ObjectStoreError>
  readonly get: (name: string) =>
    Effect.Effect<Option.Option<ObjectEntry>, ObjectStoreError>
  readonly putBlob: (meta: ObjectMeta, data: Uint8Array) => Effect.Effect<ObjectInfo, ObjectStoreError>
  readonly getBlob: (name: string) => Effect.Effect<Option.Option<Uint8Array>, ObjectStoreError>
  readonly info:   (name: string) => Effect.Effect<Option.Option<ObjectInfo>, ObjectStoreError>
  readonly list:   Effect.Effect<ReadonlyArray<ObjectInfo>, ObjectStoreError>
  readonly delete: (name: string) => Effect.Effect<void, ObjectStoreError>
  readonly watch:  (options?: { readonly includeHistory?: boolean }) =>
    Stream.Stream<ObjectInfo, ObjectStoreError>
  readonly seal:   Effect.Effect<void, ObjectStoreError>
  readonly status: Effect.Effect<ObjectStoreStatus, ObjectStoreError>
}

export interface ObjectEntry {
  readonly info: ObjectInfo
  readonly data: Stream.Stream<Uint8Array, ObjectStoreError>   // digest-verified; fails, never truncates silently
}

The one genuinely subtle upstream shape is get()'s result: the SDK returns { info, data: ReadableStream, error: Promise<Error | null> } — the error promise resolves after the stream ends, reporting digest mismatches or missing chunks that could not be known upfront. We fold that side-channel into the data stream itself: consume the bytes, then await the error promise, and fail the stream if it reports. An Effect stream that completed successfully therefore means the digest checked out — the type does what the SDK's documentation asks the caller to remember to do. Links (link/linkStore) are deferred; they're cross-bucket pointers with niche semantics, reachable via the escape hatch.

14. Services: request/reply, institutionalized

Any process can answer requests with §2's primitives: subscribe to a subject under a queue group, respond to each message. Do that at fleet scale and operational questions follow — what services exist? are they alive? how many requests has each endpoint handled, how slow, how many errors? The NATS services framework (a.k.a. micro) is a convention that answers these without new server features: a service registers its endpoints as ordinary queue-group subscriptions, and additionally listens on well-known control subjects ($SRV.PING, $SRV.INFO, $SRV.STATS, each scoped by name and instance id) where it answers discovery probes with standardized JSON. Errors travel back to callers in two reserved headers, Nats-Service-Error and Nats-Service-Error-Code. Like Kafka minus… no — like a service mesh's control plane, minus the mesh: discovery, health, and stats ride the same connection as the traffic.

The SDK's builder is imperative (svc.addEndpoint("echo", handler) with callbacks or iterators, mutating a started service). An Effect service definition should instead be a value — declared once, run scoped, handlers as effects with dependencies in R. This is the same move HttpApi makes over raw route registration:

// NatsMicro.ts
export const make: <R>(options: ServiceOptions<R>) =>
  Effect.Effect<RunningService, NatsError, NatsClient | Scope.Scope | R>
export const layer: <R>(options: ServiceOptions<R>) =>
  Layer.Layer<never, NatsError, NatsClient | R>      // runs for the layer's lifetime

export type ServiceOptions<R> = {
  readonly name: string                                // e.g. "inventory"
  readonly version: string                             // semver, reported on $SRV.INFO
  readonly description?: string
  readonly metadata?: Readonly<Record<string, string>>
  readonly queue?: string                  // @default "q"
  readonly endpoints: Readonly<Record<string, Endpoint<R>>>
}

export type Endpoint<R> = {
  readonly handler: (msg: NatsMessage.NatsMessage) =>
    Effect.Effect<Payload | void, EndpointError, R>
  readonly subject?: string                // @default group-prefixed endpoint name
  readonly queue?: string
  readonly metadata?: Readonly<Record<string, string>>
  readonly concurrency?: number | "unbounded"   // @default 1 (serial, SDK-equivalent)
}

export class EndpointError extends Schema.TaggedErrorClass<EndpointError>(
  "effect-nats/NatsMicro/EndpointError"
)("EndpointError", {
  code: Schema.Int,          // → Nats-Service-Error-Code header
  description: Schema.String // → Nats-Service-Error header
}) {}

The runtime loop per endpoint: the wrapper takes the SDK's iterator mode (no callback), consumes it as a stream, and runs the handler per message with the configured concurrency. A returned Payload becomes msg.respond(payload); a returned void means the handler responded manually (or chose not to); a failed EndpointError becomes respondError(code, description); a defect becomes respondError(500, "internal error") plus a logged cause — the caller gets a well-formed error either way, and the service never crashes on a bad request. The R parameter is the quiet headline: handlers declare dependencies (a database service, a config), and they surface in the layer's requirements, type-checked end to end. Stats (num_requests, num_errors, processing_time) are maintained by the SDK and stay accurate because we respond through its ServiceMsg paths.

The consumer side is client, wrapping the SDK's discovery scatter-gather (which is just §9's requestMany against the control subjects):

export const client: Effect.Effect<MicroClient, never, NatsClient>
export interface MicroClient {
  readonly ping:  (options?: DiscoverOptions) => Stream.Stream<ServiceIdentity, NatsError>
  readonly info:  (options?: DiscoverOptions) => Stream.Stream<ServiceInfo, NatsError>
  readonly stats: (options?: DiscoverOptions) => Stream.Stream<ServiceStats, NatsError>
}

Priced trade-off: declaring endpoints as a record forfeits the SDK's ability to add endpoints to a live service at runtime, and nested dynamic groups get flattened into explicit subject strings. Both are the cost of endpoints-as-values; a team that genuinely needs runtime mutation drops to the escape hatch (RunningService.service exposes the SDK object). We judge static definitions the right default for the same reason Effect judges layers the right default: topology you can read is topology you can test.

16. Non-goals

Out of scope, and why

Deno — excluded per requirements; nothing in the layering blocks a future DenoConnector (upstream ships @nats-io/transport-deno with the identical factory shape). Push consumers, batch/FastIngest publishing, DirectStreamAPI, scheduled publishes, ObjectStore links — deferred within their sections (§11.5, §13) with the escape hatch as the interim path. Typed subject routing (schema-derived subject hierarchies à la RPC router) — tempting, genuinely novel, and premature until the plain surface is proven; NatsMicro is where it would grow. Auth callout / advanced nkey flows beyond the four Auth variants — reachable via transformOptions with a raw authenticator.
